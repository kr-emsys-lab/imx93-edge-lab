# Target EdgeLock Enclave (ELE) Hardware Interface Guide

This document maps out the NXP hardware-specific integration points, SDK paths, and key management architecture required to implement the `--engine ele` runtime verification engine on the i.MX93 platform.

---

## 1. Yocto Project SDK Compilation Environment

When cross-compiling the C++ application or configuring an external tool/AI agent (like Devin AI) to interface with the hardware libraries, use the absolute sysroot paths below generated by your Yocto Scarthgap (6.6) environment.

### Core Sysroot Include Path
The core HSM enclave headers (`hsm_api.h`, type definitions, structure bindings) must be explicitly resolved against this location:
/opt/fsl-imx-xwayland/6.6-scarthgap/sysroots/armv8a-poky-linux/usr/include/hsm

Your CMake architecture must append this include directory when the target compilation flag `ENABLE_IMX_ELE` is active.

---

## 2. Key Management & Provisioning Matrix

To execute hardware-isolated validation, both the decryption key and the signature verification key are bound inside the enclave context.

| Key Asset | Target Subservice | Handling Mechanism |
| :--- | :--- | :--- |
| **AES-256 Symmetric Key** | `hsm_cipher_one_go` | Loaded into an ELE Key Slot (Transient RAM slot via wrapped key blob or pre-provisioned slot ID). |
| **ECDSA Public Key** | `hsm_verify_signature` | Anchored inside the ELE Keystore (Persistent NVM file structure or transient RAM slot). |

### Provisioning Framework Strategies

#### Strategy A: Persistent NVM Provisioning (Hardened)
Persistently write the keys into the ELE secure filesystem wrapper inside Linux. 
1. Format the raw host-generated public key into NXP’s required key container structure.
2. Synchronize the tracking files with the Linux encrypted key store directory structure (interacting with `/etc/keys/` or `/var/` based on platform architecture policies).
3. Track and request the assigned permanent Key Slot ID during initialization.

#### Strategy B: Ephemeral RAM Key Import (Fallback/Volatile Track)
If persistent non-volatile storage modifications are constrained by lifecycle states:
1. Export the key from your host infrastructure wrapped inside a localized transport key.
2. At application execution, read the encrypted blob from the filesystem and invoke `hsm_open_key_store_service`.
3. Perform a secure key injection command to map the asset directly to an isolated volatile RAM slot handle.
